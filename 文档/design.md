# 数脉 ShuMai 设计文档

## 1. 文档目的

本文档面向 `shumai-v7-1.jsx` 的维护者与扩展开发者，说明该文件的整体设计思想、架构组织、核心数据模型、状态流、交互机制、页面职责和后续演进建议。

该设计文档描述的对象包括：

- React 源码文件：`shumai-v7-1.jsx`
- 编译后的单文件运行版本：`index.html`

---

## 2. 系统定位

“数脉 ShuMai” 是一个围绕中考数学复习设计的单文件智能学习系统。

它不是一个通用题库壳，而是一个带有明确方法论的学习产品，强调：

- 以知识图谱组织内容
- 以依赖关系解释错误根因
- 以四模块推动复习节奏
- 以题目、方法、知识点三者互相映射
- 以 AI 补充个性化解释、诊断与预测

---

## 3. 设计目标

### 3.1 产品目标

- 为中考数学复习提供完整闭环
- 降低学生在海量题目中迷失的概率
- 让“做错题”自动回到“补知识”
- 让“不会复习”转化为“有路径地补差”

### 3.2 技术目标

- 单文件可交付
- 数据内置，便于离线演示
- UI 自包含，不依赖外部样式框架
- 支持本地状态持久化
- 可扩展 AI 能力

### 3.3 设计约束

- 不依赖复杂工程结构
- 尽可能不引入第三方 UI 库
- 页面逻辑通过组件与状态组织，而非完整路由系统
- 大量业务数据直接嵌入源码

---

## 5. 核心数据模型设计

## 5.1 知识点模型 `TOPICS`

知识点是本系统最核心的业务实体。

知识点不仅用于展示，还承担：

- 题目归类
- 方法映射
- 图谱节点展示
- 掌握状态记录
- 依赖关系追因
- 学习路径规划

一个知识点通常包含：

- `id`：唯一标识
- `name`：知识点名称
- `domain`：领域归属，如代数 / 几何 / 统计
- `freq`：考查频率
- `examYears`：出现年份列表
- `totalScore`：累计分值
- 位置 / 图谱展示信息
- 依赖或前置关系信息

设计价值：

- 以知识点作为全系统锚点，统一连接“题目、方法、掌握状态、诊断结果”。

## 5.2 方法模型 `METHODS`

方法不是孤立内容，而是与知识点绑定。

每个方法对象包含：

- `id`
- `name`
- `cat`
- `desc`
- `topics`
- `example`

设计价值：

- 让学生看到“这类题靠什么方法解决”
- 让题目训练不止停留在结果层，而是进入方法层

## 5.3 真题模型 `EXAM_QS`

真题对象是系统中最大的数据集之一。

通常字段包括：

- `id`
- `yr`
- `no`
- `city`
- `type`
- `topic`
- `subTopics`
- `score`
- `diff`
- `methods`
- `content`
- `answer`
- `sol`
- `error`
- `upLink`
- `downLink`

设计价值：

- 题目不仅是练习材料，还是诊断和推荐的证据来源
- 通过 `subTopics` 实现复合题知识点拆解
- 通过 `methods` 实现题目与解法体系互联

## 5.4 晨读卡模型 `MORNING_DATA`

为知识点提供轻量记忆结构：

- `slogan`
- `trap`

设计价值：

- 强化高频口诀式记忆
- 降低晨读页的认知负担

## 6. 状态管理设计

## 6.1 全局状态

全局状态主要集中在根组件 `App`。

关键状态包括：

- `view`
- `detailId`
- `sidebarOpen`
- `mastered`
- `wrongSet`
- `basicWrongSet`
- `aiModel`
- `dsKey`
- `dbKey`
- `showKeyInput`
- `showClearConfirm`

这些状态统一由根组件维护，原因是：

- 多页面共享
- 页面切换后仍需保留
- 需要集中持久化

## 6.2 局部状态

每个页面包含自身局部交互状态，例如：

- `PagePractice` 的筛选器状态
- `PageWrong` 的展开解析状态和语音状态
- `PageDiag` 的加载态和诊断结果
- `PagePredict` 的年份和预测结果
- `PageMethods` 的分类筛选
- `PageModules` 的当前模块与知识点选择

设计原则：

- 跨页面共享的状态放到根组件
- 页面内部短生命周期状态留在页面内部

## 6.3 持久化策略

持久化入口：

- `loadStorage()`
- `saveStorage(patch)`

使用的存储键：

- `shumai_v7`

持久化内容：

- 掌握状态
- 错题本状态
- 基础错题本状态
- 模型选择
- API Key

设计优点：

- 结构集中
- 易维护
- 切换页面和刷新后可恢复学习进度

风险点：

- 单个键承载多类数据，未来字段增长时需要版本管理

## 7. 页面设计与职责划分

## 7.1 `PageHome`

职责：

- 作为系统首页和总览仪表盘
- 汇总掌握进度、知识域分布、快捷入口
- 提供错题本、智能诊断等高频入口

设计特点：

- 信息密度高
- 适合作为导航中枢
- 强调“下一步该做什么”

## 7.2 `PageMorning`

职责：

- 提供轻量复习入口
- 基于知识口诀与易错点进行晨读

设计特点：

- 节奏轻
- 更偏记忆与快速回顾
- 独立于高强度刷题流程

## 7.3 `PageGraph`

职责：

- 展示知识图谱结构
- 支持节点点击进入知识详情

设计特点：

- 使用纯 React 计算图布局
- 不依赖 D3
- 兼顾交互性和单文件可移植性

## 7.4 `PageModules`

职责：

- 落地“四模块学习法”
- 将学习过程拆成不同训练强度阶段

设计特点：

- 是整个学习闭环的主干页面
- 既能补基础，又能做进阶和压轴

## 7.5 `PageDetail`

职责：

- 呈现单知识点详情
- 连接上下游依赖、练习入口和掌握状态

设计价值：

- 作为知识图谱与题库的桥梁页面

## 7.6 `PageMethods`

职责：

- 浏览 23 种解题方法
- 根据分类查看方法说明和适用知识点

设计价值：

- 建立“题目之上”的方法意识

## 7.7 `PagePractice`

职责：

- 真题筛选与作答练习
- 错题收录与解析展开

设计价值：

- 承担题库主入口职责
- 是错题沉淀的主要来源

## 7.8 `PageWrong`

职责：

- 聚合错题
- 追溯错因根源
- 给出补差路径
- 提供语音讲解

设计价值：

- 将“错题列表”升级为“错误分析中心”

## 7.9 `PageDiag`

职责：

- 生成学情诊断
- 汇总薄弱知识点
- 给出学习路径
- 输出 AI 报告

设计价值：

- 从静态数据展示走向个性化分析

## 7.10 `PagePredict`

职责：

- 利用历年数据生成中考预测

设计价值：

- 提供复习优先级建议
- 增强产品的策略感

## 8. 关键能力设计

## 8.1 知识依赖追因 `traceRootCause`

这是系统中最关键的业务函数之一。

输入：

- `topicId`
- `masteredSet`

输出：

- `unmastered`
- `roots`

设计逻辑：

- 从当前知识点开始
- 沿前置依赖向下搜索
- 找出未掌握链路
- 从中识别最底层的根节点

被使用于：

- 错题本
- 智能诊断
- 推荐学习路径

设计意义：

- 避免“只在表层知识点反复练习”
- 把补差指向根因而不是表象

## 8.2 AI 调用抽象 `callAI`

系统没有把 AI 写死为某一服务，而是定义统一入口。

输入：

- `system`
- `user`
- `history`

输出：

- 文本结果

当前支持：

- DeepSeek
- 豆包

设计意义：

- 便于后续切换模型
- 页面逻辑无需关心具体供应商差异

注意事项：

- API Key 保存在本地存储
- 需提醒用户自行保管密钥

## 8.3 语音能力

通过两级策略实现：

- 优先在线 TTS 音频
- 失败则回退浏览器原生朗读

设计意义：

- 提高兼容性
- 避免单一朗读方案不可用时整项功能失效

## 9. 交互流设计

## 9.1 学习闭环

推荐的核心交互流是：

- 首页进入
- 选择知识图谱或四模块
- 学知识 / 做题
- 错题进入错题本
- 错题本触发根因追溯
- 智能诊断生成学习路径
- 回到知识点详情或模块继续补差

## 9.2 题目到知识点的回流

当学生做错题时，不应停在“答案是什么”。

系统通过：

- `topic`
- `subTopics`
- `traceRootCause`

把一道题回流到知识点和前置根因。

## 9.3 方法体系嵌入

题目和知识点都能关联方法。

这意味着学习不是：

- 只背结论

而是：

- 知识点 -> 题型 -> 方法 -> 迁移应用

---

## 10. 响应式设计

断点策略：

- Mobile: `< 640px`
- Tablet: `640 - 1023px`
- Desktop: `>= 1024px`

通过：

- `useWindowSize()`
- `BPCtx`
- `useBP()`

进行响应式控制。

设计原则：

- 根布局适配移动端导航
- 页面内部通过 `isMobile` 控制排版、网格和按钮尺寸

---

## 11. 安全与配置设计

## 11.1 API Key

API Key 通过界面输入，并保存在本地。

优点：

- 用户使用方便
- 支持多模型切换

风险：

- 浏览器本地存储不适合高敏感密钥长期保存

建议：

- 如后续上线公网版本，应使用后端中转并避免前端直存密钥

## 11.2 本地单文件运行

`index.html` 可双击打开，适合演示和单机使用。

优点：

- 部署成本低
- 分发方便

局限：

- 不适合大型团队协作开发
- AI 相关接口仍受浏览器网络环境限制

## 12. 当前实现中的技术债

### 12.1 单文件过大

问题：

- `shumai-v7-1.jsx` 集成了过多职责
- 阅读、测试、维护成本高

影响：

- 小改动也容易引发连锁影响
- 难以做精细化复用

### 12.2 数据与视图强耦合

问题：

- 题库和页面逻辑混在同一文件

影响：

- 不利于数据版本管理
- 不利于独立校验题库质量

### 12.3 缺少错误边界

问题：

- 个别页面运行时异常可能导致整块界面黑屏

建议：

- 增加 React Error Boundary
- 给 AI 和 TTS 功能增加更清晰的失败提示

### 12.4 数据质量警告

构建过程中已发现少量对象存在重复键：

- `answer`
- `sol`

影响：

- 后面的值会覆盖前面的值
- 可能导致题目解析与答案不一致

建议：

- 编写数据校验脚本
- 对 `EXAM_QS` 做结构化清洗

- `utils/tts.js`
- `utils/trace.js`

可逐步引入：

- 自定义 hooks
- context 分层
- 或轻量状态管理

目标：

- 降低根组件负担
- 提升页面复用性

## 13.3 数据治理

建议增加：

- 字段模式校验
- 重复键检测
- 数据导入导出机制

## 13.4 部署能力增强

建议后续加入：

- 正式 `package.json`

## 14. 面向维护者的结论

`shumai-v7-1.jsx` 的最大特点不是“React 页面很多”，而是它把：

- 知识结构
- 方法体系
- 真题数据
- 学习闭环
- AI 分析
- 本地离线运行

全部压缩进了一个可交付的学习系统原型中。

它适合：

- 快速演示
- 单文件交付
- 产品概念验证
- 离线展示

它不适合长期维持为单文件形态继续膨胀。

因此，对这个系统最合理的维护策略是：

- 短期：继续稳定单文件版本，修数据、补错误边界、提升文档完整度
- 中期：拆分模块，建立正式工程化结构
- 长期：将 AI、题库、诊断能力做服务化或独立数据层治理
