# V4.72 SKE/PET 后台操作日志持久化方案

日期：2026-05-09

## 目标

V4.70 和 V4.71 已经完成后台 Skill 人工操作的前端确认感：最近人工操作、改写前后对比、保存后字数变化。

但这些记录仍然只存在当前页面状态中，刷新后会清空。V4.72 先不急着写代码，而是把真正的后台操作日志持久化方案设计清楚，避免把审计、推荐效果、学生行为事件混在一张表里。

## 设计原则

- 操作日志只记录管理员行为，不替代学生行为事件。
- 操作日志只做追踪、回看、排查，不自动改变推荐算法。
- 不记录密钥、token、手机号明文、AI 提示词全文快照以外的敏感信息。
- Skill 改写需要保留前后摘要或 diff，但要避免日志表变成长期大文本仓库。
- 数据库变更必须走独立 migration，不继续扩大 `server/schema.sql` 的混合风险。
- 上线前必须先本地验证，再申请服务器和数据库授权。

## 建议表结构

表名：`admin_operation_logs`

建议字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `id` | `bigserial primary key` | 日志 ID |
| `admin_user_id` | `integer` | 操作管理员 ID，可为空以兼容早期 token 缺失 |
| `admin_role` | `text` | 操作时角色快照 |
| `action` | `text not null` | 操作类型，如 `skill_weight_adjust`、`skill_rewrite` |
| `target_type` | `text not null` | 目标类型，如 `prompt_skill` |
| `target_id` | `integer` | 目标 ID |
| `target_key` | `text` | Skill key 或业务 key |
| `target_name` | `text` | 操作时名称快照 |
| `before_value` | `jsonb` | 操作前摘要 |
| `after_value` | `jsonb` | 操作后摘要 |
| `reason` | `text` | 操作原因，可由系统建议带入 |
| `source` | `text` | 来源，如 `pet_watchlist`、`skill_row` |
| `request_id` | `text` | 后续排查链路用，可为空 |
| `created_at` | `timestamptz default now()` | 创建时间 |

建议索引：

- `created_at desc`
- `(target_type, target_id, created_at desc)`
- `(action, created_at desc)`
- `(admin_user_id, created_at desc)`

## API 草案

### 写入日志

`POST /api/admin/operation-logs`

只允许 admin 调用。

请求体建议：

```json
{
  "action": "skill_rewrite",
  "targetType": "prompt_skill",
  "targetId": 12,
  "targetKey": "math_common_hint_first_step_001",
  "targetName": "先找突破口",
  "beforeValue": {"contentLength": 82, "contentPreview": "先不要急着算..."},
  "afterValue": {"contentLength": 96, "contentPreview": "先圈出题目问什么..."},
  "reason": "后台观察清单建议改写",
  "source": "pet_watchlist"
}
```

### 读取日志

`GET /api/admin/operation-logs?action=&targetType=&targetId=&limit=`

默认返回最近 50 条，最大 200 条。

前端展示建议：

- Skill 管理页保留“最近人工操作”，优先读持久化日志。
- 没有接口或数据库未迁移时，继续降级使用 V4.70 的前端临时记录。
- Skill 详情或观察清单卡片可展示“最近一次人工操作”。

## 与现有系统关系

不复用 `prompt_skill_events`：

- `prompt_skill_events` 是学生端/学伴端行为事件，服务推荐效果评估。
- `admin_operation_logs` 是后台人工操作，服务审计、回看和排查。
- 两者可以通过 `target_id` / `skill_key` 在后台聚合，但不混表。

不改 `server/schema.sql`：

- 新增独立 migration：`server/migrations/20260509_admin_operation_logs.sql`
- 本地实现和验证完成后，再决定是否提交后端代码
- 服务器执行 SQL 必须单独授权

## 上线步骤建议

1. 新增 migration 草案和本地验证脚本，不执行 SQL。
2. 新增 service/API，数据库不可用时返回空列表或 `stored:false`。
3. 前端“最近人工操作”优先读持久化日志，失败时保留本地临时记录。
4. 本地 `node --check`、`npm run build`、`git diff --check`。
5. 创建干净 commit，不混入 `server/schema.sql`、deploy、TTS / ASR、study-plan、动画。
6. 如需上线，另开服务器授权任务：备份数据库、执行 migration、部署后端、验证 API。

## 风险与边界

- 本阶段只做方案，不写 migration，不写 API，不执行 SQL。
- 不连接服务器。
- 不部署。
- 不操作 PM2 / Nginx / PostgreSQL。
- 不触碰 `server/schema.sql`、`deploy/`、TTS / ASR、study-plan、public animations、dist-singlefile、local-notes、`.env*`。
