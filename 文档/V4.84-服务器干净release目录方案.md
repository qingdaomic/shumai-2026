# V4.84 服务器干净 release 目录方案

日期：2026-05-09

## 目标

本阶段只设计干净 release 目录方案，不连接服务器，不创建目录，不拉代码，不安装依赖，不部署。

V4.83 已经完成 `/opt/shumai` 与 `/var/www/shumai` 的上线前备份，并确认原 `/opt/shumai` 不适合直接 `git pull`。V4.84 的目标是把下一次服务器实操拆成一条不覆盖原目录、可验证、可回滚的 release 路线。

## 当前问题

服务器当前线上状态：

- `/opt/shumai` 属主为 `root:root`
- `/opt/shumai/.git` 属主为 `root:root`
- `/var/www/shumai` 属主为 `www-data:www-data`
- root PM2 中 `shumai-api` 在线
- `shumai-api` 当前运行路径为 `/opt/shumai/server/index.js`
- `ubuntu` 用户 PM2 列表为空
- 服务器当前 `HEAD` 落后 GitHub 最新 50 个提交
- 服务器工作区有 43 个 tracked 脏改、23 个未跟踪项
- 后台操作日志相关文件与服务器脏改重叠

因此，不建议在 `/opt/shumai` 原目录内执行：

- `git pull`
- `git reset`
- `git checkout`
- 直接覆盖文件
- 直接部署前端
- 直接重启 PM2

## 推荐目录结构

建议新增独立 release 根目录：

```text
/opt/shumai-releases/
  v4.84-YYYYMMDD-HHMMSS/
    repo/
      package.json
      src/
      server/
      dist/
```

保留现状：

```text
/opt/shumai/              # 当前线上目录，暂不覆盖
/var/www/shumai/          # 当前线上前端静态目录，暂不覆盖
/home/ubuntu/shumai-backups/v4.83-20260509-173223/
```

后续可选软链，但不在第一步执行：

```text
/opt/shumai-current -> /opt/shumai-releases/v4.84-YYYYMMDD-HHMMSS/repo
```

## V4.85 可执行步骤草案

### 1. 创建 release 目录

只创建新目录，不改旧目录：

```bash
sudo mkdir -p /opt/shumai-releases
sudo chown ubuntu:ubuntu /opt/shumai-releases
RELEASE=/opt/shumai-releases/v4.85-$(date +%Y%m%d-%H%M%S)
mkdir -p "$RELEASE"
```

### 2. 拉取 GitHub 最新代码

只 clone 到新目录：

```bash
git clone https://github.com/qingdaomic/shumai-2026.git "$RELEASE/repo"
cd "$RELEASE/repo"
git checkout main
git rev-parse HEAD
git status -sb
```

要求：

- `git status -sb` 必须干净
- `HEAD` 必须等于当时 GitHub `origin/main`
- 不从 `/opt/shumai` 复制源码文件到 release 目录

### 3. 前端安装与构建验证

```bash
cd "$RELEASE/repo"
npm ci
npm run build
test -f dist/index.html
du -sh dist
```

要求：

- `npm ci` 成功
- `npm run build` 成功
- `dist/index.html` 存在
- 不复制 `dist` 到 `/var/www/shumai`
- 不 reload Nginx

### 4. 后端依赖验证

```bash
cd "$RELEASE/repo/server"
npm ci --omit=dev
node --check index.js
```

要求：

- 后端依赖安装成功
- `node --check index.js` 成功
- 不启动 3001
- 不重启 PM2
- 不读取或打印环境变量

### 5. 后台操作日志文件确认

```bash
cd "$RELEASE/repo"
test -f server/migrations/20260509_admin_operation_logs.sql
test -f server/services/admin-operation-logs.js
grep -n "operation-logs" server/api/admin.js
grep -n "operation-logs" src/App.jsx
```

要求：

- migration 文件存在
- service 文件存在
- 后端 API 已挂载
- 前端已调用后台操作日志接口

## 后续切换方案

### 前端切换

前端切换必须在构建通过后另行执行：

```bash
sudo rsync -a --delete "$RELEASE/repo/dist/" /var/www/shumai/
sudo nginx -t
sudo systemctl reload nginx
```

回滚：

- 使用 V4.83 备份中的 `/var/www/shumai` 恢复
- 或保留切换前 `/var/www/shumai` 快照，再 rsync 回去

### 后端切换

当前 `server/ecosystem.config.cjs` 的 `cwd` 写死为 `/opt/shumai/server`，不能直接用于 release 目录切换。

建议下一阶段生成临时 PM2 配置，不改仓库文件，例如：

```js
module.exports = {
  apps: [{
    name: 'shumai-api-v4.85',
    script: 'index.js',
    cwd: '/opt/shumai-releases/v4.85-YYYYMMDD-HHMMSS/repo/server',
    instances: 1,
    exec_mode: 'fork',
    node_args: '--experimental-vm-modules',
    env: {
      NODE_ENV: 'production',
      PORT: 3002
    }
  }]
};
```

先用 3002 做旁路健康检查：

```bash
sudo pm2 start /tmp/shumai-api-v4.85.config.cjs --only shumai-api-v4.85
curl http://localhost:3002/api/health
sudo pm2 stop shumai-api-v4.85
sudo pm2 delete shumai-api-v4.85
```

注意：

- 3002 旁路验证仍然需要生产环境变量，必须避免打印密钥
- 不与当前 3001 线上服务抢端口
- 不保存 PM2 列表，除非正式切换

### 数据库 migration

后台操作日志 migration 必须在代码 release 验证之后、正式后端切换之前执行：

```bash
psql "$DATABASE_URL" -f "$RELEASE/repo/server/migrations/20260509_admin_operation_logs.sql"
```

前置要求：

- 先做 PostgreSQL 备份
- 不打印 `DATABASE_URL`
- 只执行这一份 migration
- 不修改 `server/schema.sql`
- migration 成功后记录表结构和索引

## 回滚原则

如果 release 验证失败：

- 删除新建 release 目录即可
- 不改 `/opt/shumai`
- 不改 `/var/www/shumai`
- 不改 PM2
- 不执行 SQL

如果前端切换后失败：

- 从切换前快照或 V4.83 备份恢复 `/var/www/shumai`
- `nginx -t` 通过后 reload

如果后端切换后失败：

- 停止新 PM2 进程
- 恢复 root PM2 当前 `shumai-api`
- 保留失败日志
- 不做破坏性数据库回滚，除非单独授权
