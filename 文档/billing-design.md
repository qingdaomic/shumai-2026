# 中考数学系统收费与套餐体系设计文档

> 版本：V1.0 | 对应阶段：V2 正式运营版

---

## 一、设计目标

收费体系需要满足：

- 支持灵活的套餐配置，可随时调整价格和权益
- 订单和支付记录完整可追溯
- 支持主流支付渠道（微信支付、支付宝）
- 支持手动赋权（管理员直接赋予学员套餐）
- 试用期体验后平滑转化付费
- 退款流程清晰，管理端可操作

---

## 二、套餐体系设计

## 三、订单生命周期

### 3.1 订单状态流转

```
学员选择套餐
     │
     ▼
created（订单已创建）
     │
     ▼
pending_payment（等待支付）
     │
     ├── 支付成功 ──► paid（已支付）
     │                   │
     │                   ├── 正常到期（系统处理）
     │                   │
     │                   └── 退款申请 ──► refunded（已退款）
     │
     ├── 支付超时 ──► closed（已关闭）
     │
     └── 支付失败 ──► failed（支付失败）
```

### 3.2 支付成功后处理

1. 订单状态改为 `paid`，记录 `paid_at`
2. 计算套餐到期时间：`expires_at = paid_at + duration_days`
3. 更新 `student_profiles.plan_id` 和 `plan_expires_at`
4. 更新 `users.status = active`
5. 记录订单到 `billing_records`（可选附加明细）

### 3.3 套餐到期处理

定时任务每天凌晨检测：

1. 找出所有 `plan_expires_at < now()` 且 `status = active` 的用户
2. 批量更新 `users.status = expired`
3. 可选：发送到期提醒通知

## 四、支付渠道接入规划

### 4.1 V2 阶段接入建议

| 渠道 | 优先级 | 说明 |
|------|--------|------|
| 微信支付 | 高 | 覆盖最广，必接 |
| 支付宝 | 高 | 必接 |
| 人工打款 | 中 | 管理端手动赋权，用于特殊情况 |

### 4.2 支付安全规范

- 前端只负责展示，不传递支付状态到后端
- 支付结果以支付渠道服务端回调为准，不信任前端
- 回调必须验签
- 同一订单支付成功回调做幂等处理（防重复处理）
- 支付相关接口必须 HTTPS

### 4.3 退款规则建议

- 付费 7 天内且使用不超过 30 道题，支持全额退款
- 超出以上条件，不支持退款（可配置）
- 退款操作在管理端由管理员手动发起
- 退款成功后：订单状态改 `refunded`，学员状态改 `expired`

## 五、手动赋权流程

适用于以下场景：

- 内部测试账号
- 合作渠道赠送
- 活动兑换
- 退款后重新补偿

**操作路径**：管理端 → 学员详情 → 赋予套餐

**字段**：

- 选择套餐（从 plans 中选）
- 有效天数（可自定义，不一定按套餐默认天数）
- 备注（记录赋权原因）

**系统处理**：

- 更新 `student_profiles.plan_id` 和 `plan_expires_at`
- 更新 `users.status = active`
- 生成一条特殊订单记录（payment_method = manual，amount_fen = 0）

## 六、财务统计需求

管理端财务汇总提供以下数据（仅 admin 可见）：

| 指标 | 说明 |
|------|------|
| 今日收款 | 今日 paid 订单金额合计 |
| 本月收款 | 本月 paid 订单金额合计 |
| 累计收款 | 全部 paid 订单金额合计 |
| 各套餐销量 | 每个套餐的已支付订单数 |
| 退款总额 | 全部 refunded 订单金额合计 |
| 付费转化率 | paid 学员数 / 总注册学员数 |

## 七、V1.5 阶段收费实现建议

V1.5 阶段不需要做完整支付接入，建议先做：

- 套餐表基础配置
- 管理员手动赋权功能
- 学员端展示套餐详情页（静态，无支付按钮）
- 前端根据 `status` 和 `plan_expires_at` 做功能限制

真正的支付接入放在 V2 阶段，上线前需要完成：

- 营业执照
- 微信支付/支付宝商户号注册
- 支付回调地址配置
- 完整的支付测试和安全验收

## 八、核心设计决策

| 决策 | 选择 | 原因 |
|------|------|------|
| 金额存储单位 | 分（整数） | 避免浮点精度问题 |
| 套餐变更策略 | 以最新赋权为准，旧套餐覆盖 | 简单可控 |
| 并发支付 | 同一学员同一时间只允许一个 pending_payment 订单 | 防止重复下单 |
| 价格调整 | 套餐调价不影响已付费的历史订单 | 历史订单冗余保存 plan_name 和 amount_fen |
| 免费内容 | 晨读卡和知识图谱浏览不做任何限制 | 降低体验门槛 |
