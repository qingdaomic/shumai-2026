# V4.28 后端稳定能力审查报告

> 阶段：树脉 V4.28｜后端稳定能力逐文件审查与安全提交计划
> 日期：2026-05-08
> 窗口：2号窗口，本地与服务器共同执行任务
> 边界：本阶段只审查和制定计划；不提交、不推送、不部署、不服务器操作、不重启 PM2、不改数据库、不删除文件。

## 1. 总体结论

V4.28 对 V4.27 归档出的剩余后端栈做了逐文件审查。结论比 V4.27 更收束：

- 当前仍不适合直接做一个“后端整体提交”。
- V4.29 可以做一个很窄的“后端稳定能力安全提交”，但只应包含不依赖新 schema、不挂载 TTS / ASR、不启动动画 cron、不改部署脚本的后端增量。
- 大部分看起来有价值的能力都需要拆小：学习计划、微信上下文、家长端目标建议、每日任务读取计划、管理端资源统计、SVG 文件缓存，都依赖新表或新字段。
- `server/index.js` 当前最危险，因为它同时挂载了 TTS、ASR、study-plan，并在服务启动时调用动画 cron。这个文件不能原样进入 V4.29。
- `server/schema.sql` 不能混进后端功能提交，必须走独立数据库迁移审查。

建议 V4.29 先做“低风险后端稳定提交”，只纳入非常有限的文件或文件片段；随后再开 V4.30 做 schema 迁移计划。

这些文件有安全部分，但当前 diff 混入了 schema、TTS / ASR、动画或部署风险，不能整体提交。

### `server/index.js`

当前混合内容：

- 挂载 `ttsRouter`、`asrRouter`、`studyPlanRouter`。

审查判断：

## 9. 是否建议先做后端提交，还是进入 SKE-1

建议先做 V4.29 的小后端提交，再进入 SKE-1。

原因：

- 当前仍有少量低风险后端文案和教师建议能力可以先沉淀。
- 如果直接进入 SKE-1，`server/schema.sql` 会继续变大，并和现有 `study_plans`、`tts_usage`、`question_svgs` 等 schema 变化混在一起。
- 先把 V4.29 做窄，可以让后续 SKE-1 的数据库结构和 API 更干净。

推荐顺序：

1. V4.29：后端低风险稳定提交计划执行。
2. V4.30：数据库 / schema 迁移拆分计划。
3. SKE-1：教学 Skill 引擎 schema 草案、种子 Skill 与最小 API。
