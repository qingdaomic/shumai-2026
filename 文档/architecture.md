# 中考数学系统总体架构设计

## 1. 架构目标

本架构设计用于支撑系统从“单文件前端原型”演进为“可部署、可运营、可扩展的学习平台”。

架构设计需要同时满足两类目标：

- 短期：快速上线前端测试版，支持学生试用
- 长期：平滑接入后端、用户体系、内容运营、支付与管理端

## 2. 总体架构原则

### 2.1 前后端解耦

前端不能长期直接依赖硬编码数据结构，必须逐步引入 service 层和统一数据接口，以便未来替换为真实 API。

### 2.2 先可用，再扩展

优先保障学员端的核心学习体验；对于收费、运营、审核等复杂系统，先做结构预留，再逐步落地。

### 2.3 管理端与学员端分角色设计

学员端重体验与学习流程；管理端重内容管理、审核、统计和运营配置。两者共享底层服务，但前端与权限应隔离。

### 2.4 数据资产中心化

题库、知识点、视频、行为日志、订单等都应在后端统一管理，不能长期散落在前端本地存储中。

## 3. 目标架构分层

### 3.1 展示层

- 学员端 Web
- 管理端 Web
- 可选移动端 H5 / 小程序

### 3.2 应用层

- 用户认证服务
- 学习内容服务
- 题库服务
- 诊断服务
- 错题与学习记录服务
- 视频资源服务
- 支付与订单服务
- 管理后台服务
- 行为日志服务

### 3.3 数据层

- 关系数据库
- 缓存
- 对象存储
- 日志存储
- 检索或推荐辅助数据

### 3.4 第三方服务

- AI 接口服务
- B站等视频平台链接
- 短信/邮件服务
- 支付渠道
- 监控告警平台

## 4. 阶段化架构

## 4.1 阶段一：前端测试版

特点：

- 学员端先上线
- 主要内容仍可由本地数据或模拟服务提供
- 部分数据使用 localStorage 或浏览器缓存
- 所有业务访问统一通过前端 service 层

目标：

- 快速验证核心学习流程
- 为后端接入预留接口形态

### 4.2 阶段二：轻后端接入

新增：

- 注册登录
- 学员档案
- 学习记录同步
- 错题记录持久化
- 基础管理端
- 基础行为日志

目标：

- 支撑真实用户持续使用
- 避免只靠浏览器本地存储

### 4.3 阶段三：正式运营架构

新增：

- 完整后台服务
- 支付与套餐系统
- 视频资源审核与推荐
- 内容管理中心
- 数据分析和运营看板
- 更细的权限控制

目标：

- 支撑收费、增长、教研和内容运营

## 5. 推荐模块边界

### 5.1 学员端

负责：

- 页面展示
- 学习流程交互
- 用户主动操作
- 基础本地状态管理
- 调用服务接口

不负责：

- 核心业务规则最终判断
- 权限决定
- 收费状态最终校验
- 资源审核状态管理

### 5.2 管理端

负责：

- 内容维护
- 用户管理
- 视频审核
- 数据查看
- 配置变更

### 5.3 后台服务

负责：

- 身份鉴权
- 数据持久化
- 状态流转
- 审核规则
- 推荐与统计逻辑
- 订单、支付、日志等核心业务能力

## 6. 核心数据流

### 6.1 学习数据流

- 学员进入知识点
- 查看定义、方法、视频
- 完成基础题或题组训练
- 产生练习结果
- 写入错题/学习记录
- 触发诊断结果更新
- 学员端展示新的学习建议

### 6.2 内容管理数据流

- 管理员新增或修改知识点/题目/视频
- 系统记录审核状态
- 发布后进入学员端可见范围
- 用户行为反哺运营与内容优化

### 6.3 视频资源数据流

- 系统采集候选视频
- 管理端审核
- 关联知识点
- 设置主视频和备选视频
- 学员端查看和点击
- 后台记录点击与资源状态

## 7. 前端到后端的演进要求

为了避免后续大规模重构，当前前端测试版应遵循以下约束：

- 页面尽量不要直接耦合底层常量数据
- 抽象 `topicService`、`questionService`、`diagnosisService`、`videoService`
- 统一响应结构
- 将用户态、学习记录、错题状态收敛到可替换的存储层
- 对需要鉴权的功能预留登录态判断

## 8. 部署建议

### 8.1 测试版部署

可先采用：

- 静态前端部署
- 单独域名或子域名
- 简单访问统计

### 8.2 正式版部署

推荐拆分：

- 前端站点
- 管理端站点
- API 服务
- 数据库
- 文件/对象存储
- 定时任务与监控

## 9. 风险与约束

### 9.1 纯前端长期运行风险

- 数据不同步
- 登录和付费无法可靠实现
- 学习记录易丢失
- 管理端难以真正运营

### 9.2 内容维护风险

- 单文件维护成本高
- 数据和 UI 耦合严重
- 知识点、题库、视频等资源不利于分角色协作

### 9.3 第三方资源风险

- 外部视频链接可能失效
- AI 接口成本与稳定性需要控制
- 支付与个人信息需要合规处理

## 10. 推荐结论

总体策略建议为：

- 当前阶段先上线学员端测试版
- 代码层面同步做后端接入预留
- 下一阶段优先补最小后端
- 随着用户量和业务复杂度增长，再逐步完善管理端、支付、日志和运营体系
