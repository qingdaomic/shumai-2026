# V4.31 数据库变更拆分计划

> 阶段：树脉 V4.31｜schema / 数据库变更拆分计划
> 日期：2026-05-08
> 窗口：2号窗口，本地与服务器共同执行任务
> 边界：本阶段只检查、分类、制定计划；不提交、不 push、不部署、不服务器操作、不重启 PM2、不改数据库、不执行 SQL、不删除文件。

## 1. 总体结论

V4.31 重点审查 `server/schema.sql` 以及依赖它的后端能力。当前结论是：现有 schema diff 不是一个单一功能变更，而是把学习计划、用户角色、微信上下文、管理端资源统计、TTS 用量记录等多条能力线混在一起。

因此，当前不建议把 `server/schema.sql` 作为一个整体提交，更不建议直接同步到服务器或依赖后端启动时的 `initDb()` 自动改线上库。

更稳的做法是：

- SKE-1 单独设计 `prompt_skills` 与 `prompt_skill_events`，不要复用当前混合 schema diff。
- 学习计划、微信上下文、管理端资源统计、TTS / ASR、SVG 缓存分别拆成独立 migration。
- 上线前先备份线上 PostgreSQL，再执行可回滚、可审计的迁移脚本。

## 3. schema 变更逐项分类

### 3.1 品牌注释

变更：

- `-- 数脉数据库建表脚本` 改为 `-- 树脉数据库建表脚本`

分类：

- 文档 / 品牌统一

风险：

- 无数据库行为风险。

建议：

- 可随任意安全 schema 文档提交进入，但没有必要单独上线。

### 3.2 `study_plans` 学习目标计划表

当前新增：

- `study_plans`
- `idx_study_plan_active`
- `idx_study_plan_user`

字段包括：

- `user_id`
- `subject`
- `stage`
- `current_score`
- `target_score`
- `target_date`
- `daily_minutes`
- `plan_data`
- `status`
- `created_at`
- `updated_at`

分类：

- 每日任务 / 学习计划相关
- 反向规划保存闭环

依赖 API：

- `server/api/study-plan.js`
- `server/services/daily.js`
- `server/api/parent.js`
- `server/bot.js`
- `server/index.js` 中 `/api/study-plan` 路由挂载

上线风险：

- 如果只迁移表、不上线 API，前端无法保存反向规划。
- 如果上线 API、但表未迁移，`/api/study-plan`、每日任务、家长端摘要和微信今日任务都会 500。
- `ON CONFLICT (user_id, subject) WHERE status = 'active'` 依赖部分唯一索引，migration 必须确保索引存在。

建议：

- 单独形成“学习计划数据库 + API”阶段。
- 可以作为 SKE 之前的学习闭环基础，但不是 SKE-1 必需。
- 上线前写单独 migration：`migrations/YYYYMMDD_add_study_plans.sql`。

### 3.3 `users.exam_date`

当前新增：

- `ALTER TABLE users ADD COLUMN IF NOT EXISTS exam_date DATE`

分类：

- 每日任务 / 学习计划相关
- 微信提醒 / 考前冲刺相关

依赖 API / 服务：

- `server/api/study-plan.js` 保存 `targetDate` 后同步写入 `users.exam_date`
- `server/bot.js` 中 `getSprintPhase(user.exam_date)` 用于考前提醒

上线风险：

- 字段本身低风险，但它属于学习计划和考前提醒的共享字段。
- 如果仅新增字段，不改变业务逻辑，风险较低。

建议：

- 与 `study_plans` 放在同一 migration 或学习计划前置 migration。
- 不建议与 TTS / ASR / 动画混在一起。

### 3.4 `users.role`

当前新增：

- `ALTER TABLE users ADD COLUMN IF NOT EXISTS role VARCHAR(20) DEFAULT 'student'`

分类：

- 家长端 / 教师端 / 管理端权限相关

依赖 API：

- `server/api/admin.js`
- `server/api/parent.js`
- `server/api/teacher.js`
- 现有 admin 兜底也支持 `ADMIN_PHONE` 环境变量

上线风险：

- 字段低风险，但权限语义高风险。
- 若线上没有 `role` 字段，部分教师端 / 家长端 / 管理端接口会依赖失败。
- 默认 `student` 合理，但管理员、教师、家长角色仍需要后台或 SQL 设置，不应在本阶段直接改生产数据。

建议：

- 单独做“用户角色字段迁移”。
- migration 只增加字段，不批量改用户角色。
- 管理员角色分配应另开权限配置任务。

### 3.5 `users.wechat_context_token` 与 `users.wechat_last_seen`

当前新增：

- `ALTER TABLE users ADD COLUMN IF NOT EXISTS wechat_context_token TEXT`
- `ALTER TABLE users ADD COLUMN IF NOT EXISTS wechat_last_seen TIMESTAMPTZ`

分类：

- 微信上下文相关

依赖 API / 服务：

- `server/api/wechat.js`
- `server/bot.js`

上线风险：

- 需要重启 ClawBot 才能生效。
- `wechat_context_token` 会影响主动推送上下文，必须微信实测。
- 如果 bot 逻辑上线但字段未迁移，收到微信消息或定时推送时会数据库报错。

建议：

- 单独形成“微信上下文持久化”阶段。
- 先迁移字段，再上线 `server/api/wechat.js` 和 `server/bot.js` 对应逻辑。
- 上线后观察绑定、解绑、私聊回复、定时推送是否稳定。

### 3.6 `system_config.resource_billing`

当前新增：

- `system_config` 中新增 key：`resource_billing`

分类：

- 管理端资源 / 计费提醒相关
- TTS 资源包统计相关

依赖 API：

上线风险：

- 本身只是配置 seed，但 `server/api/admin.js` 同时读取服务器资源、PM2 状态和 TTS 用量。

## 4. SKE-1 最小必需数据库范围

当前 `server/schema.sql` 的未提交 diff 里没有 SKE-1 的核心表。

如果进入 SKE-1，建议最小数据库只包含两张表：

### `prompt_skills`

用途：

- 保存可推荐、可启停、可评估的教学 Skill。

建议字段：

- `id SERIAL PRIMARY KEY`
- `code VARCHAR(80) UNIQUE NOT NULL`
- `title VARCHAR(100) NOT NULL`
- `subject VARCHAR(30) DEFAULT 'math'`
- `stage VARCHAR(30) DEFAULT '初中'`
- `topic_code VARCHAR(80)`
- `method_code VARCHAR(80)`
- `scene VARCHAR(50) NOT NULL`
- `trigger_tags JSONB DEFAULT '[]'`
- `prompt TEXT NOT NULL`
- `example_questions JSONB DEFAULT '[]'`
- `weight SMALLINT DEFAULT 50`
- `status VARCHAR(20) DEFAULT 'active'`
- `created_at TIMESTAMPTZ DEFAULT NOW()`
- `updated_at TIMESTAMPTZ DEFAULT NOW()`

### `prompt_skill_events`

用途：

- 记录 Skill 展示、点击、调用、反馈和效果数据。

建议字段：

- `id SERIAL PRIMARY KEY`
- `user_id INT REFERENCES users(id)`
- `skill_id INT REFERENCES prompt_skills(id)`
- `event_type VARCHAR(30) NOT NULL`
- `question_id VARCHAR(80)`
- `question_type VARCHAR(30)`
- `topic_code VARCHAR(80)`
- `payload JSONB DEFAULT '{}'`
- `created_at TIMESTAMPTZ DEFAULT NOW()`

建议索引：

- `idx_prompt_skills_lookup(subject, stage, topic_code, scene, status)`
- `idx_prompt_skill_events_user(user_id, created_at DESC)`
- `idx_prompt_skill_events_skill(skill_id, created_at DESC)`

SKE-1 不需要：

- `study_plans`
- `tts_usage`
- `resource_billing`
- `wechat_context_token`
- `wechat_last_seen`
- SVG 文件缓存改造
- 动画 cron

## 5. 建议 migration 策略

不建议继续只依赖 `server/index.js` 启动时执行整份 `server/schema.sql`。

原因：

- `schema.sql` 已经混合了基础表、ALTER、seed、索引和未来能力表。
- 后端启动时自动执行整份 schema，容易把未准备好的表和配置带到线上。
- 线上数据库需要备份、审计、可回滚，而不是“启动时顺手改”。

建议新增目录：

- `server/migrations/`

建议拆分：

1. `20260508_add_prompt_skills.sql`
   - SKE-1 必需。
   - 只建 `prompt_skills` 和 `prompt_skill_events`，可带 30-50 条种子 Skill。
2. `20260508_add_study_plans.sql`
   - 只做学习计划表和 `users.exam_date`。
3. `20260508_add_user_roles.sql`
   - 只加 `users.role`。
4. `20260508_add_wechat_context.sql`
   - 只加 `wechat_context_token`、`wechat_last_seen`。
5. `20260508_add_tts_usage.sql`
   - 只加 `tts_usage`。
6. `20260508_seed_resource_billing.sql`
   - 只 seed 管理端资源提醒配置。

每个 migration 要满足：

- 可重复执行：使用 `IF NOT EXISTS`。
- 不写真实密钥、账号、token。
- 上线前先备份。
- 配套说明“依赖哪些 API，回滚影响是什么”。

## 7. 必须暂缓的数据库变化

暂缓项：

- `study_plans`，除非下一阶段明确做学习计划闭环。
- `wechat_context_token` / `wechat_last_seen`，除非下一阶段明确做微信上下文实测。
- `resource_billing`，除非下一阶段明确做管理端资源统计安全审查。
- `tts_usage`，除非下一阶段明确做 TTS / ASR 上线审查。
- SVG 文件缓存 API，不建议和 SKE-1 混在一起。

暂缓原因：

- 它们都依赖未提交或未部署的 API。
- 多数需要后端重启。
- 部分涉及服务器资源、云服务成本、文件系统或微信 bot 行为。
- 当前 2核2G 服务器应继续保持后端上线边界干净。

## 8. 上线数据库风险

当前存在以下风险：

- 如果直接部署 `server/index.js`，会同时挂载 TTS、ASR、study-plan，并启动动画 cron。
- 如果直接执行整份 `server/schema.sql`，会把多个阶段的数据库能力一起带入线上。
- 如果只上线 API 不迁移 schema，学习计划、微信上下文、TTS 统计、管理端资源统计会报错。
- 如果只迁移 schema 不审查 API，线上数据库会出现暂时不用的表和配置，增加维护成本。
- `users.role` 涉及权限；不能在没有角色分配策略的情况下批量修改线上用户。
- TTS / ASR 与资源统计涉及用量成本；不能和普通后端能力混上。
